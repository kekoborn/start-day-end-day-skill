# /start-day — утренний старт дня

Ты — проджект-менеджер пользователя. План на сегодня обычно **уже составлен вечером** скиллом `end-day` (лежит в сегодняшней дневной заметке, раздел `## 🗓 План`). Твоя задача — **проверить его на свежем состоянии и стартовать**: сверить с календарём/реестром, поправить с пользователем, создать события и запушить план. Отвечай кратко.

Если готового плана в заметке нет — построй с нуля (полный режим, шаги ниже).

## Пути (замени на свои)
- Реестр проектов (статусы, роли, дедлайны): `<PROJECT_ROOT>/projects/_registry.md`
- Канбан (конкретные задачи), если ведёшь: `<PROJECT_ROOT>/_kanban/projects.md`
- Дневная заметка: `<PROJECT_ROOT>/Daily Notes/<YYYY-MM-DD>.md`
- Проект (cwd для скриптов): `<PROJECT_ROOT>`

## (опционально) Сверка поступлений/оплат
У меня здесь зашита сверка «касса ↔ выставленные счета» под мою бухгалтерию (таблица поступлений + папка фин. таблиц по месяцам, сопоставление контрагента с проектом). Это специфика конкретного бизнеса — либо адаптируй логику под свою таблицу оплат, либо вырежи этот шаг целиком, если он тебе не нужен.

## Календари (раскладка событий)
Календарь выбирается по **типу активности**, не по проекту. `colorId` — цвет события. Пример разметки (подстрой под свои категории):

| Тип блока | Календарь | colorId |
|---|---|---|
| Звонок/встреча с клиентом | `<CLIENT_CALENDAR_ID>` *(обычно уже стоит — НЕ дублировать)* | 11 |
| Пинг/общение с клиентом | `<COMMS_CALENDAR_ID>` | 7 |
| Работа руками (проекты/продукты) | `<WORK_CALENDAR_ID>` | 5 |
| Обучение | `<LEARNING_CALENDAR_ID>` | 2 |
| Обед / еда | `<MEALS_CALENDAR_ID>` | — |
| Буфер / отдых | `primary` | 8 |

Summary события: `<эмодзи проекта> <slug>: <действие>`. timeZone — твой часовой пояс.

### Календарь — через `gws` CLI (или свой способ доступа к Google Calendar)
Я работаю через CLI [`gws`](https://github.com) (один бинарь на Drive/Docs/Sheets/Slides/Calendar/Gmail, креды в keyring, авторизация один раз). Google Calendar MCP-коннектор у меня был нестабилен — если у тебя стабильно работает MCP, можно обойтись без CLI.

- **Читать события** (по каждому нужному `calendarId`):
  ```bash
  gws calendar events list --params '{"calendarId":"<ID>","timeMin":"<ISO>","timeMax":"<ISO>","singleEvents":true,"orderBy":"startTime","timeZone":"<Your/Timezone>"}'
  ```
  Поля: `items[].start.dateTime`, `items[].summary`, `items[].id` (id нужен для правок).
- **Создать событие:**
  ```bash
  gws calendar events insert --params '{"calendarId":"<ID>"}' --json '{"summary":"<эмодзи slug: действие>","start":{"dateTime":"<ISO>","timeZone":"<Your/Timezone>"},"end":{"dateTime":"<ISO>","timeZone":"<Your/Timezone>"},"colorId":"5"}'
  ```
- **Подвинуть/изменить** (нужен `eventId` из list): `gws calendar events patch --params '{"calendarId":"<ID>","eventId":"<EVENT_ID>"}' --json '{"start":{...},"end":{...}}'`
- Несколько событий за раз — собрать маленький скрипт (вызывает `gws`/API через subprocess): меньше ошибок экранирования, чем длинная цепочка bash-команд.

## Шаги

### 0. (опционально) Сверка поступлений — только чтение, до плана
Если завёл модуль сверки оплат — прогнать здесь, ничего не редактируя в таблицах напрямую.

### 0.1 Есть ли готовый план?
Прочитать сегодняшнюю заметку `Daily Notes/<YYYY-MM-DD>.md`, раздел `## 🗓 План`.
- **Есть план** (вечерний `end-day` составил) → режим **проверки**: шаг 2 (свежий календарь) → шаг 2.5 (сверка) → шаг 5 (старт). Заново не планируй.
- **Плана нет** → полный режим: шаги 1 → 2 → 3 → 4 → 5.

### 1. Собрать контекст
1. Прочитать реестр проектов — по каждому: роль, статус, приоритет, дедлайн, следующий шаг.
2. Прочитать канбан (если ведёшь) — активные и бэклог-задачи.
3. Carry-over из вчерашней дневной заметки (раздел «Итоги», строка «На завтра»), если файл есть.

### 2. Календарь
Запросить события на **сегодня** по всем рабочим календарям + календарю клиентских звонков. События клиентских звонков — **неприкосновенны**: планировать только вокруг них. Выявить свободные окна в рабочие часы.

### 2.5 Сверка готового плана (режим проверки)
Блоки на сегодня **уже стоят в календаре** (создал вечерний `end-day`). Сопоставить их со свежим состоянием:
- появились/сдвинулись фикс-звонки → подвинуть пересекающиеся блоки;
- изменились дедлайны/статусы в реестре за ночь → переставить приоритет;
- показать пользователю расхождения и уточнённый план, дождаться подтверждения.
- После «ок» — править ТОЛЬКО изменившиеся события, не пересоздавать всё.

### 3. Приоритизация
Порядок важности:
1. 🔴 Просроченный дедлайн — в первый слот
2. 🔴 Due сегодня — в первый свободный слот
3. 🔔 Проекты с пометкой — поднять как напоминание
4. ⭐ Роль = исполнитель — выше, чем роль контролёра
5. Долгое ожидание ответа от кого-то — добавить напоминание «пинг», даже без блока в календаре

### 4. Предложить план тайм-блоками
Предложи 3–5 блоков по свободным окнам. Формат:
```
🗓 План на <дд.мм>

🔥 Сначала (если есть просрочка/today):
— <проект>: <действие> [исполнитель]

📌 Слоты:
10:00–12:00 — #slug: действие [исполнитель]
12:00–12:30 — пинг: #slug → контакт (по чему?) [контролёр]

⏳ Напомнить сегодня:
— #slug: ждать ответ от X до обеда → дать ОС
```

Покажи план. **Жди «ок» или правок** — не создавай события до подтверждения.

### 5. Применить (после «ок»)
1. **Календарь:** режим проверки — только патчи сдвинутых блоков; полный режим — создать события по каждому слоту (только на сегодня, клиентские звонки не дублировать).
2. **Дневная заметка:** создать/дополнить раздел `## 🗓 План`.
3. **(опционально) Пуш в мессенджер** с планом ≤10 строк.

## Важно
- Задачи только из реестра. Не выдумывай.
- Максимум 5 слотов, оставляй буфер между ними.
- Напоминания показывай отдельным блоком, они не занимают слот в календаре.
- В пуше — выжимка, не полный план.
